**Duração:** 3–4 dias

## Meta

## Contrato mem_save (Engram)

Payload padrão gerado pelo bridge:

```json
{
  "title": "Architecture layer: API",
  "type": "architecture",
  "topic_key": "cb-layer-api",
  "what": "42 nodes in API layer covering auth, billing, ...",
  "why": "Synced from Understand Anything knowledge graph",
  "where": ".understand-anything/knowledge-graph.json#layer:api",
  "learned": "Generated by context-bridge sync"
}
```

Prefixo `cb-` (context-bridge) em topic_keys para evitar colisão com memórias manuais.

## Critérios de aceite

- [ ] `EngramClient` usa HTTP quando `engram serve` responde
- [ ] Fallback CLI funciona quando HTTP indisponível
- [ ] `doctor` mostra qual backend está ativo
- [ ] `topic_key_from_graph_node` é estável entre runs
- [ ] Dedup evita duplicata óbvia no mesmo topic_key
- [ ] `pytest tests/test_engram*` passa offline

## Notas técnicas

- HTTP base URL default: `http://127.0.0.1:7437`
- Respeitar `ENGRAM_HTTP_TOKEN` se configurado
- Não implementar MCP direto — bridge fala REST/CLI, não stdio MCP
- Erros de rede: mensagem acionável ("start engram serve or ensure engram in PATH")

---

## Riscos

| Risco | Mitigação |
|-------|-----------|
| API Engram muda | Pin versão mínima engram na doc; testes contra fixture OpenAPI |
| serve bind 127.0.0.1 only | Documentar; CLI fallback sempre disponível |
